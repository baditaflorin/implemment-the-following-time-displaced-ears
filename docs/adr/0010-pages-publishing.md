# 10. GitHub Pages publishing strategy

- Status: accepted
- Date: 2026-05-11

## Context

Mode A requires that every `git push` to `main` updates the live site. GitHub Actions is explicitly forbidden by project constraints (account-wide Actions lock; CI must be local). Therefore the only viable strategy is the **branch-folder mode**: Pages serves the contents of a directory inside the default branch.

## Decision

- Source: **`main` branch, `/docs` folder.**
- Build tool: Vite 5 (ADR 0003).
- Vite's `build.outDir` = `docs` with `build.emptyOutDir = false`. This lets us commit ADRs at `docs/adr/` and the architecture diagram at `docs/architecture.md` alongside the built `index.html` / `assets/` without the build clobbering them.
- Vite's `base = './'` — relative asset URLs so the same bundle works at `https://baditaflorin.github.io/<repo>/` and at a future custom domain without rebuilding.
- A `docs/404.html` is committed (identical content to `index.html` after build, generated by Vite's `--ssr-manifest`-free flow via a small post-build copy). This gives SPA-style fallback if we ever add client routing.
- A `docs/.nojekyll` is committed so Pages serves files and directories that start with `_` literally and skips Jekyll processing.

## Why not a `gh-pages` branch?

A separate branch would require pushing twice (code, then built site) and would tempt people to skip rebuilding. Keeping the build output committed on `main/docs/` makes the live site a function of `HEAD`, which is easier to reason about and to roll back (`git revert HEAD`).

## Cache busting

Vite emits hashed filenames (`assets/index-Abc123.js`). The single mutable file is `index.html`. GitHub Pages serves `index.html` with a short `Cache-Control: max-age=600` by default; that's acceptable for our cadence (commits land within ~1 minute of pushing).

## Custom domain

If a custom domain is added later, the only changes are: add a `CNAME` file under `docs/`, point the DNS A/AAAA or CNAME records per https://docs.github.com/en/pages/configuring-a-custom-domain-for-your-github-pages-site, and (optionally) update the `homepage` field in `package.json`. The `base = './'` decision avoids a rebuild.

## Reconciling with `.gitignore`

`docs/` is intentionally NOT gitignored — it's the published artifact. Build cache (`.vite/`) and the intermediate `dist/` directory are gitignored. The pre-push hook verifies `docs/index.html` exists and is non-empty.

## Consequences

- Every build dirties `docs/assets/`. This is expected churn, not noise — `data:` commits per workflow conventions are acceptable here, though we'll mostly fold them into `feat:`/`fix:` commits that triggered the rebuild.
- Rollback is `git revert`, no separate Pages re-deploy step.

## Enabling Pages

After the first push, Pages is enabled via the GitHub API:

```
gh api -X POST /repos/baditaflorin/<repo>/pages \
  -f source[branch]=main -f source[path]=/docs
```

This is documented in the Makefile target `pages-enable` (run once per repo).
